To get started you need to sign up for a regular Twitter account as well as a Twitter developer account.

Twitter uses the OAuth 1.0 protocol for authentication when an app (or bot in our case) makes api calls.

The Client / Consumer = our bot. The Service Provider = Twitter. The Resource Owner = the Twitter users that our bot will be gathering tweet data from.

The OAuth protocol performs a kind of three-legged dance:

  1. The client obtains an unauthorized request token from the service provider.
  2. The resource owner authorizes the request token.
  3. The client exchanges the request token for an access token.
  4. The client uses the access token to access protected resources on behalf of the resource owner

Twitter has number of APIs, but we'll mainly concern ourselves with the REST API and the Streaming API

What is REST? REST ('Representational State Transfer) is a bit like a command line where the service is the computer. RESTful APIs enable developers to access information and resources using a simple HTTP invocation. You can obtain domain-specific data simply by pointing a URL to a specific location.

So, the Twitter Search API is a RESTful service that enables users to point to a specific URL and retrieve a variety of tweets that meet the criteria specified in the URL. This enables you, as a developer, to accept input within a Web application and dynamically query Twitter based on that input, using a simple URL that encodes the input into a format that the API understands.

Using the Twitter REST API, we can access profiles, make updates, get followers, add followers, do search.

Take a look at the https://dev.twitter.com/overview/api/ to get a breakdown of tweet attributes, user attributes
